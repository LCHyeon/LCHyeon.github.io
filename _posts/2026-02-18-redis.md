## 1. 백엔드 상태(State)와 저장소의 관점

### 1.1 상태란 무엇인가

---

백엔드에서 말하는 **상태(State)** 는 결국 [**데이터]**다.

- 사용자의 정보
- 로그인 여부
- 주문 진행 상태
- 요청 횟수
- 최근 조회 기록

모든 요청은 상태를 **읽거나 변경**한다. 즉, 백엔드는 **본질적으로 상태를 관리하는 시스템**이다.

### 1.2 상태의 유형: 영구 상태와 임시 상태

---

백엔드의 상태는 성격에 따라 다음과 같이 구분할 수 있다.

| 구분 | 영구 상태 (Persistent State) | 임시 상태 (Ephemeral State) |
| --- | --- | --- |
| 예시 | 유저 정보, 주문/결제 내역, 게시글 | 로그인 세션, 인증 토큰, 캐시, 카운터, 최근 기록 |
| 보존성 | 반드시 보존되어야 함 | 일정 시간이 지나면 사라져도 됨 |
| 데이터 손실 허용 | 허용되지 않음 | 상황에 따라 허용 가능 |
| 구조적 특성 | 관계/무결성 중요 | 단순 접근 모델 중심 |
| 접근 패턴 | 트랜잭션/조인 필요 가능 | 빠른 읽기/쓰기 중심 |
| 적합한 저장소 | 디스크 기반 데이터베이스 | 인메모리 저장소 (예: Redis) |

### 1.3 임시 상태의 관리 방식

---

임시 상태는 영구 보존이 목적이 아니지만, 요청 처리 과정에서는 매우 빈번하게 읽히고 변경된다.

문제는 다음과 같다.

- 매 요청마다 디스크 기반 DB에 접근하면 비용이 증가한다.
- 서버가 여러 대일 경우, 각 서버의 메모리에 상태를 두는 것은 공유가 어렵다.
- 세션이나 토큰처럼 “시간이 지나면 자동으로 사라져야 하는 상태”는 별도 관리가 필요하다.

따라서 임시 상태는:

- 빠르게 접근 가능하고
- 여러 서버가 공유할 수 있으며
- 수명을 제어할 수 있는

저장소에서 관리하는 것이 일반적이다.

이러한 요구를 충족하는 대표적인 시스템이 **Redis**다.

---

## 2. Redis 개요

### 2.1 Redis란 무엇인가

---

Redis는 **네트워크로 접근하는 인메모리 데이터 저장소**이며, 단순 Key-Value를 넘어 **여러 자료구조(Data Structures)** 를 Value로 제공하는 시스템이다.

> Redis = In-memory Key-Value Data Structure Server
> 

### 2.2 Redis의 사용 목적

---

- 자주 읽히는 데이터의 빠른 제공(캐시)
- 임시 상태(state)의 저장과 공유(세션/토큰/카운터 등)
- 원자적 연산이 필요한 간단한 상태 머신/카운팅
- 자료구조 기반 기능(최근 기록, 집합 연산, 랭킹, 스트림 처리)

---

## 3. Redis의 핵심 구조와 동작 특성

### 3.1 인메모리 기반과 성능 특성

---

- Redis는 기본적으로 데이터를 RAM에 저장해 빠르다.
- 디스크 기반 저장소와 비교할 때 “랜덤 접근 + 작은 연산”에서 강점이 크다.
- 단, 메모리 기반이므로 메모리 용량/정책(Eviction)에 영향을 받는다.

### 3.2 단일 스레드 이벤트 루프와 원자성

---

- Redis는 기본적으로 **단일 스레드로 명령을 처리**한다.
- 결과적으로 각 명령은 **명령 단위로 원자적(atomic)** 으로 실행된다.
- “동시성 제어” 관점에서 단일 명령의 race condition을 줄이기 쉽다.

### 3.3 Key 기반 접근과 데이터 모델

---

- Redis는 Key를 통해 값을 직접 찾는다.
- Key 하나는 **한 가지 타입의 자료구조**를 갖는다.
- 데이터 설계는 “테이블 설계”보다 “Key 설계 + 자료구조 선택”에 가깝다.

### 3.4 만료(Expiration)와 TTL

---

Redis는 Key 단위로 TTL을 설정할 수 있다.

```
EXPIRE key seconds
PEXPIRE key milliseconds
TTL key
PTTL key
PERSIST key
```

- TTL이 지나면 Key는 만료된다.
- TTL은 캐시/임시 상태에 매우 유용하다.

### 3.5 메모리 관리와 Eviction

---

Redis는 메모리 상한을 설정할 수 있고, 초과 시 정책에 따라 키를 제거한다.

핵심 키워드:

- `maxmemory`
- `maxmemory-policy` (예: allkeys-lru, volatile-ttl 등)

의미:

- Redis를 캐시로 사용할 때 “삭제될 수 있음”을 전제로 설계해야 한다.

### 3.6 영속성(Persistence) 개요: RDB / AOF

---

Redis는 메모리 기반이지만 디스크로 저장하는 옵션을 제공한다.

- **RDB**: 스냅샷 저장(특정 시점 덤프)
- **AOF**: 쓰기 명령 로그 저장(재실행으로 복구)

용도는 “캐시 중심”인지 “데이터 내구성”을 원하는지에 따라 결정된다.

---

## 4. Redis 데이터 모델링 기본

### 4.1 Key 네임스페이스와 이름 규칙

---

일반적으로 `:` 를 구분자로 사용해 네임스페이스를 만든다.

- 예: `user:1`, `session:abc`, `rate:ip:1.2.3.4`

권장 관점:

- “도메인:엔티티:식별자:속성”처럼 계층을 분명히 둔다.
- 너무 긴 Key는 관리가 어려워지므로 일관된 규칙이 중요하다.

### 4.2 Value는 “자료구조”를 가진다

---

Redis의 Value는 단순 문자열이 아니라:

- String / List / Hash / Set / Sorted Set / Stream … 같은 자료구조 중 하나다.

따라서 설계는:

- “어떤 Key를 만들까?”
- “그 Key에 어떤 자료구조를 둘까?”

두 가지 결정을 중심으로 진행된다.

### 4.3 원자적 연산과 복합 작업 묶기

---

- 단일 명령은 원자적이다.
- 여러 동작을 “논리적으로 한 번에” 처리해야 할 때는 다음 개념을 사용한다.
    - **Pipeline**: 왕복을 줄여 성능 개선(원자성 목적 아님)
    - **Transaction(MULTI/EXEC)**: 여러 명령을 묶어 실행
    - **Lua Script**: 복합 로직을 원자적으로 실행

---

## 5. Redis 데이터 타입

> 아래 시간 복잡도는 “대표 연산 기준”이며, 명령어/데이터 크기에 따라 달라질 수 있다.
> 

### 5.1 String

---

- 가장 기본 타입(바이너리 세이프 문자열)
- 주요 용도: 값 캐시, 플래그, 카운터

대표 명령:

```
SET key value
GET key
INCR key
DECR key
```

특성:

- 카운터(INCR/DECR)는 원자적으로 증가/감소한다.

### 5.2 List

---

- 순서가 있는 컬렉션(양쪽 push/pop에 강함)
- 주요 용도: 최근 기록, 단순 큐

대표 명령:

```
LPUSH key value
RPUSH key value
LPOP key
RPOP key
LRANGE key start stop
LTRIM key start stop
```

특성:

- “양끝 삽입/삭제”는 빠른 편
- “범위 조회(LRANGE)”는 읽는 개수에 비례해 비용 증가

### 5.3 Hash

---

- field-value 맵(객체 표현에 적합)
- 주요 용도: 사용자/세션/메타데이터 같은 “속성 묶음”

대표 명령:

```
HSET key field value
HGET key field
HGETALL key
HDEL key field
```

특성:

- 작은 객체를 다룰 때 메모리 효율이 좋은 편

### 5.4 Set

---

- 중복 없는 집합
- 주요 용도: 유니크 멤버 관리, 집합 연산

대표 명령:

```
SADD key member
SREM key member
SISMEMBER key member
SMEMBERS key
```

### 5.5 Sorted Set (ZSet)

---

- score로 정렬되는 집합
- 주요 용도: 랭킹, 정렬된 조회, 시간 기반 정렬

대표 명령:

```
ZADD key score member
ZRANGE key start stop
ZREVRANGE key start stop
ZREM key member
```

특성:

- 삽입/삭제는 보통 로그 시간대 비용
- 범위 조회가 강력함

### 5.6 Stream

---

- 로그(append-only) 성격의 자료구조
- 주요 용도: 이벤트 스트림, consumer group 기반 처리

대표 명령(개념 수준):

```
XADD key * field value
XREAD key
XGROUP CREATE key groupname $
XREADGROUP GROUP groupname consumername ...
```

특성:

- consumer group / ack / 재처리 같은 “스트림 처리” 기능 제공

---

## 핵심 정리

> - 백엔드는 본질적으로 **상태를 관리하는 시스템**이다.
> - 상태는 **영구 상태와 임시 상태**로 구분할 수 있다.
> - 영구 상태는 디스크 기반 DB가 적합하고, 임시 상태는 빠른 접근이 가능한 인메모리 저장소가 적합하다.
> - Redis는 **인메모리 Key-Value 데이터 구조 서버**로, 임시 상태 관리와 고속 데이터 처리를 위해 설계된 시스템이다.
> - Redis의 설계는 “테이블 중심”이 아니라 **Key 설계 + 자료구조 선택**을 중심으로 이루어진다.
