## 1. 전체 구조 개요

GitHub Actions와 AWS를 활용한 CI/CD 자동화 구조는 다음과 같이 구성된다.

```
코드 변경
   ↓
GitHubActions(CI)
   ↓
Docker 이미지 생성
   ↓
ECR 저장
   ↓
ECS / EKS 배포 (CD)
```

> 여기서 각 구성 요소는 다음 역할을 수행한다.
> 

| 구성 요소 | 역할 |
| --- | --- |
| GitHub Actions | CI 수행 및 CD 트리거 |
| Docker | 실행 가능한 이미지 생성 |
| ECR | 컨테이너 이미지 저장소 |
| ECS | 컨테이너 실행 환경 |
| EKS | Kubernetes 기반 실행 환경 |

---

## 2. GitHub Actions

### 2.1 역할

---

- CI 파이프라인 실행
- Docker 이미지 빌드
- AWS 인증 수행
- ECR push
- ECS/EKS 배포 트리거

즉, GitHub Actions는 **자동화 실행 엔진**이다.

### 2.2 장단점

---

| 구분 | 내용 |
| --- | --- |
| **장점** | GitHub 저장소와 완전 통합, 설정 단순(YAML 기반), 별도 서버 운영 불필요, Marketplace 액션 활용 가능, 빠른 도입 가능 |
| **단점** | GitHub 플랫폼 종속, 복잡한 엔터프라이즈 파이프라인 구성에는 한계 가능, 러너 리소스/실행 시간 제한, 내부망 환경에는 제약 |

---

## 3. AWS 구성 요소

### 3.1 ECR (Elastic Container Registry)

---

> Docker 이미지를 저장하는 레지스트리
> 

**장점**

- AWS IAM과 통합
- ECS/EKS와 자연스럽게 연동

**단점**

- AWS 종속
- 퍼블릭 레지스트리에 비해 생태계 활용성 낮음

### 3.2 ECS (Elastic Container Service)

---

> 컨테이너 실행 및 서비스 운영
> 

**장점**

- 설정 비교적 단순
- Kubernetes 지식 불필요
- Fargate 사용 시 서버 관리 불필요

**단점**

- Kubernetes 생태계 대비 확장성 한계
- 멀티 클라우드 전략에는 부적합

### 3.3 EKS (Elastic Kubernetes Service)

---

> Kubernetes 기반 컨테이너 오케스트레이션
> 

**장점**

- 높은 확장성
- 클라우드 네이티브 표준 준수
- 다양한 배포 전략 가능

**단점**

- 초기 설정 난이도 높음
- 운영 복잡도 증가

## 4. CI 스크립트 구조 (GitHub Actions)

GitHub Actions는 YAML 파일로 정의된다.

```
.github/workflows/ci.yml
```

### 4.1 기본 CI 예시

```yaml
name: CI

on:
  push:
    branches: ["main"]
  pull_request:

jobs:
  build:
    runs-on: ubuntu-latest

    steps:
      - name: 코드 체크아웃
        uses: actions/checkout@v4

      - name: JDK 설정
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: 17

      - name: Gradle 실행 권한 부여
        run: chmod +x gradlew

      - name: 빌드 및 테스트
        run: ./gradlew clean build
```

- 구성 설명
    
    | 항목 | 설명 |
    | --- | --- |
    | `on` | 트리거 조건 |
    | `jobs` | 실행 단위 |
    | `runs-on` | 실행 환경(OS) |
    | `steps` | 순차 실행 작업 |
    | `uses` | 외부 액션 사용 |
    | `run` | 쉘 명령 실행 |

## 5. CD 스크립트 예시 (ECR + ECS 기준)

### 5.1 Docker 이미지 빌드 및 ECR Push

```yaml
- name: AWS 로그인
  uses: aws-actions/configure-aws-credentials@v4
  with:
    role-to-assume: ${{ secrets.AWS_ROLE }}
    aws-region: ap-northeast-2

- name: ECR 로그인
  uses: aws-actions/amazon-ecr-login@v2

- name: Docker 이미지 빌드
  run: |
    docker build -t my-app:${{ github.sha }} .
    docker tag my-app:${{ github.sha }} \
      ${{ steps.login-ecr.outputs.registry }}/my-app:${{ github.sha }}

- name: ECR Push
  run: |
    docker push ${{ steps.login-ecr.outputs.registry }}/my-app:${{ github.sha }}
```

## 6. CI/CD 자동화 설계 시 고려사항

- 환경 분리(dev / stage / prod)
- 이미지 태그 전략 (latest 지양, commit SHA 활용)
- 롤백 전략
- 배포 승인 단계 도입 여부
- 헬스체크 자동 검증

---

## 핵심 정리

> - GitHub Actions는 CI 실행 엔진이자 CD 트리거 역할을 수행한다.
> - Docker는 배포 가능한 실행 단위를 생성한다.
> - ECR은 컨테이너 이미지를 저장한다.
> - ECS/EKS는 컨테이너를 실행하는 환경이다.
> - CI는 검증 자동화, CD는 배포 자동화이며 두 단계는 하나의 파이프라인으로 연결된다.
> - AWS 배포 시 OIDC 기반 인증이 권장된다.
