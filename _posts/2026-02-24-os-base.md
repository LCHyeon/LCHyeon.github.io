## 1. 운영체제의 구조

### 1.1 OS란 무엇인가

---

운영체제(OS)는 하드웨어와 애플리케이션 사이에서 자원을 관리하는 시스템 소프트웨어다.

좁은 의미에서 OS는 커널(Kernel)을 의미한다.

넓은 의미에서는 커널과 기본 사용자 프로그램을 포함한 배포판을 의미한다.

<aside>

- Linux는 커널
- Ubuntu는 Linux 커널을 포함한 OS
- Windows XP는 자체 커널을 포함한 OS

</aside>

### 1.2 커널과 사용자 공간

---

> 시스템 메모리는 커널 영역(Kernel Space)을 포함한 OS 사용자 영역(User Space)으로 구분된다.
> 

![[출처 : https://rlaehddnd0422.tistory.com/152] ](attachment:e34cf71e-7376-47e0-8c13-9983f242defd:image.png)

[출처 : https://rlaehddnd0422.tistory.com/152] 

- 커널은 하드웨어를 직접 제어하는 특권 코드다.
- 사용자 공간은 일반 애플리케이션이 실행되는 영역이다.
- 커널만이 CPU, 메모리, 파일 시스템, 네트워크 장치 등에 직접 접근할 수 있다.

### 1.3 시스템 콜(System Call)

---

> 사용자 공간 프로그램은 하드웨어에 직접 접근할 수 없다. 반드시 커널을 통해 자원을 요청해야 한다.
이때 사용하는 인터페이스가 시스템 콜이다.
> 

<aside>

동작 흐름

1. 사용자 프로그램이 시스템 콜 실행
2. CPU가 user mode에서 kernel mode로 전환
3. 커널이 요청 처리
4. 작업 완료 후 다시 user mode로 복귀

즉, 커널 모드 전환은 시스템 콜을 통해 발생한다.

</aside>

## 2. 프로세스 격리와 자원 관리

### 2.1 프로세스란 무엇인가

---

프로세스는 실행 중인 프로그램이다. 각 프로세스는 독립적인 메모리 공간을 가진다. 운영체제는 다음을 관리한다.

- CPU 스케줄링
- 메모리 할당
- 파일 접근 제어
- 네트워크 자원 사용

이러한 관리 기능 덕분에 여러 프로그램이 동시에 안전하게 실행될 수 있다.

### 2.2 격리(Isolation)의 중요성

---

운영체제가 제공하는 격리는 다음을 가능하게 한다.

- 한 프로그램의 오류가 다른 프로그램에 영향 주지 않음
- 보안 유지
- 멀티 사용자 환경 지원

이 격리 개념이 이후 가상화 기술의 기반이 된다.

---

## 3. 가상화의 등장 배경과 VM

### 3.1 물리 서버 중심 구조의 한계

---

가상화 이전에는 하나의 물리 서버에 하나의 운영체제를 설치하고 애플리케이션을 실행했다.

```
Hardware
  ↓OS
  ↓Application
```

이 구조의 문제점:

- 서버 자원 활용 비효율
- 애플리케이션 간 강한 결합
- 장애 발생 시 전체 시스템 영향
- 확장 어려움

이러한 한계를 해결하기 위해 가상화 기술이 등장했다.

### 3.2 VM: OS 단위 가상화

---

VM은 하이퍼바이저(VirtualBox 등)가 만든 가상 컴퓨터다.

각 VM은 자체 운영체제를 가진다.

```
Hardware
  ↓Hypervisor
  ↓VM
  ↓GuestOS
  ↓Application
```

격리 단위는 운영체제다.

### 3.3 VM이 해결한 문제

---

- 하나의 물리 서버에서 여러 OS 실행 가능
- OS 단위 격리
- 서버 자원 활용도 개선
- 환경 독립성 확보

### 3.4 VM의 한계

---

- 각 VM마다 OS 포함
- 메모리/디스크 사용량 증가
- 부팅 및 실행 무거움

---

## 4. Container: 프로세스 단위 가상화

> **핵심 아이디어** : 운영체제 전체를 복제하지 않고, 커널을 공유한 채 프로세스를 격리하자.
> 

VM은 운영체제를 단위로 격리했지만, 컨테이너는 프로세스를 단위로 격리한다.

```
Hardware
  ↓
Host OS (Linux Kernel)
  ↓Container Runtime
  ↓Container (Process)
```

컨테이너는 VM처럼 Guest OS를 포함하지 않으며, 커널을 공유하는 격리된 프로세스일 뿐이다.

### 4.1 Docker와 Container Runtime

---

Docker는 컨테이너를 생성하고 실행하기 위한 플랫폼이다.

- Image: 실행 환경 템플릿
- Container: 이미지를 기반으로 실행된 프로세스
- Container Runtime: 컨테이너를 실제로 실행시키는 프로그램

즉,

- Image는 설계도
- Container는 실행 중인 인스턴스
- Runtime은 그 인스턴스를 띄우는 엔진

Docker는 이 전체 과정을 하나의 도구로 묶어 제공한다.

### 4.2 격리 단위의 진화

---

가상화 기술은 격리 단위를 점점 더 작게 만들며 발전해왔다.

- 물리 서버 시대: 격리 단위 = 서버
- VM 시대: 격리 단위 = 운영체제
- 컨테이너 시대: 격리 단위 = 프로세스

격리 단위가 작아질수록 자원 효율성과 배포 속도가 개선되었다.

이 흐름 위에서 MSA 확산과 함께 증가한 컨테이너를 대규모로 관리하기 위한 기술로 **Kubernetes**가 등장한다.

---

## **Kubernetes를 둘러싼 구조적 의문 정리** 
> Kubernetes를 실습하기 전, 드는 의문에 대해 LLM을 통해 생성한 답변들입니다.

- Q1. 쿠버네티스는 SSH 접속 없이 배포가 된다고 하는데, 그게 무슨 말이야?
    - **전통적인 배포는 SSH로 서버에 접속해 직접 명령을 실행했지만,
    쿠버네티스는 SSH 대신 API를 통해 “원하는 상태”를 제출하는 방식이기 때문이다.**
    
    과거 방식:
    
    ```
    CI/CD → SSH 접속 → docker pull → 서버 재시작
    ```
    
    쿠버네티스 방식:
    
    ```
    kubectl apply → Kubernetes API (HTTPS)
                      ↓
                  노드 에이전트(kubelet)가 실행
    ```
    
    즉, 사람이 서버에 들어가 명령을 치는 방식이 아니라 **클러스터가 스스로 상태를 맞추는 구조**다.
    
- Q1-1. 그럼 쿠버네티스 없으면 왜 SSH로 배포했어? HTTPS 쓰면 안 됐어?
    - **서버에는 배포를 받아줄 HTTP API가 없었기 때문이다.**
    
    배포는 OS 레벨 작업을 포함한다:
    
    - docker pull
    - 프로세스 종료
    - 새 컨테이너 실행
    - 환경 변수 적용
    
    이건 단순한 HTTP 요청이 아니라 **운영체제 명령 실행**이다.
    
    그래서:
    
    ```
    CI/CD → SSH 접속 → 서버에서 OS 명령 실행
    ```
    
    이 구조가 필요했다.
    
    Kubelet이라는 에이전트 내에 docker 실행 로직이 포함되어 있어서, SSH 접속 로직 자체의 작성 필요 없이, 쿠버네티스로 docker 이미지 버전만 제출하면 됨 → 이 방식이 API로 이뤄지고.
    
- Q1-2. GitHub에서 HTTPS로 clone하는 것과 SSH로 clone하는 것은 뭐가 달라?
    - **데이터를 전송하는 채널과 인증 방식이 다르고, Github에서 두 방식 모두 지원할 뿐.**
    
    ### HTTPS clone
    
    - HTTP 서버와 통신
    - 토큰 인증
    - Git 전용 HTTP endpoint 사용
    
    ### SSH clone
    
    - SSH 서버와 통신
    - 공개키 인증
    - 쉘이 아니라 `git-upload-pack` 같은 Git 프로그램 실행

---

- Q2. 쿠버네티스도 결국에, OS위에서 돌아가는 프로세스일텐데 직접 인스턴스 안에 프로그램을 설치하는 방법도 있을거라고 생각돼. EKS, NKS 같은 매니지먼트를 사용하는것과 직접 설치하는 것의 차이는 무엇이며, 판단 지점은 어디일까?
    - **Control Plane 운영 책임이 누구에게 있느냐가 나뉜다.**
    
    ### 기술적 차이
    
    쿠버네티스는 Linux OS 위에서 실행되는 소프트웨어이므로, EC2 같은 인스턴스에 직접 설치할 수 있다.
    
    - **직접 설치**
        
        → Control Plane + Worker Node를 모두 직접 구성·운영
        
        → etcd, API 서버, HA 구성, 업그레이드, 장애 대응까지 전부 책임
        
    - **Managed 서비스**
        - Amazon EKS
        - Naver Cloud Platform Kubernetes Service
        
        → Control Plane을 클라우드가 대신 운영
        
        → 사용자는 Worker Node와 애플리케이션만 관리
    
- Q.2-1 **Control Plane**과 **Worker Node 등**을 포함하는 쿠버네티스의 기본 구조가 뭐야?
    - 쿠버네티스는 “클러스터” 구조이며, 크게 **Control Plane**과 **Worker Node**로 나뉜다.
    
    ```
                    ┌────────────────────┐
                    │   Control Plane     │
                    │────────────────────│
                    │  API Server        │
                    │  Scheduler         │
                    │  Controller Manager│
                    │  etcd              │
                    └────────────────────┘
                               │
            ───────────────────┼───────────────────
                               │
            ┌──────────────────┴──────────────────┐
            │                                      │
    ┌──────────────────┐                 ┌──────────────────┐
    │   Worker Node 1  │                 │   Worker Node 2  │
    │──────────────────│                 │──────────────────│
    │ kubelet          │                 │ kubelet          │
    │ container runtime│                 │ container runtime│
    │ Pods             │                 │ Pods             │
    └──────────────────┘                 └──────────────────┘
    ```
    
    **[Control Plane 구성 요소]**
    
    | 구성요소 | 역할 | 설명 |
    | --- | --- | --- |
    | **API Server** | 클러스터 요청의 입구 | kubectl, CI/CD, 외부 클라이언트의 모든 요청을 받는다. 클러스터 상태 변경은 반드시 API Server를 통해 이루어진다. |
    | **etcd** | 상태 저장소 | 클러스터의 모든 상태 정보를 저장하는 Key-Value 데이터베이스. 어떤 Pod가 떠 있는지, 어떤 리소스가 존재하는지 등의 정보를 보관한다. |
    | **Scheduler** | Pod 배치 결정 | 새로 생성된 Pod를 어떤 Worker Node에 배치할지 결정한다. 리소스, 조건, 정책 등을 고려한다. |
    | **Controller Manager** | 상태 조정(Control Loop) | 사용자가 선언한 “원하는 상태”와 실제 상태를 계속 비교한다. 부족하면 생성하고, 초과하면 제거한다. |
    
    **[Worker Node 구성 요소]**
    
    | 구성요소 | 역할 | 설명 |
    | --- | --- | --- |
    | **kubelet** | 노드 에이전트 | Control Plane과 통신하는 프로세스. “이 Pod를 실행하라”는 명령을 받아 실제 실행을 관리하고 상태를 보고한다. |
    | **Container Runtime** | 컨테이너 실행 엔진 | containerd, CRI-O 등의 런타임이 실제로 이미지를 pull하고 컨테이너를 실행한다. |
    | **Pod** | 애플리케이션 실행 단위 | 쿠버네티스에서 배포의 최소 단위. 하나 이상의 컨테이너를 포함할 수 있으며, 실제 애플리케이션이 실행되는 단위다. |
